# 指示: ドキュメント自動更新（毎日実行）

## 目的

Claude（設計レビュー・実装指示担当）が常に最新のコード状態を把握できるよう、
`@workspace` でコード全体を走査し、各ドキュメントを現在の実装と同期させる。

**実行タイミング:** 毎日1回、または大きな機能追加・リファクタリングの後

---

## 更新対象ドキュメント一覧

| ファイル | 更新内容 |
|----------|----------|
| `docs/file-map.md` | ファイルの追加・削除・役割変更を反映 |
| `docs/architecture.md` | データモデル・API・認証・画面構成を実装に同期 |
| `docs/roadmap.md` | 完了済みタスクに ✅ を付ける |

---

## 各ファイルの更新ルール

---

### docs/file-map.md

#### 追加・削除の検出
- 前回から新規追加されたファイルをツリーに追加し、1行説明を付ける
- 削除されたファイルをツリーから除去する
- ファイルの役割が大きく変わった場合は説明を書き直す

#### 重点ファイルの詳細メモを最新化
以下のファイルは props / emit / 主要関数名を実際のコードから読み取り、
`## 重点ファイルの詳細メモ` セクションを更新すること:

- `apps/frontend/pages/games/[id]/play.vue`
  - 主な状態変数名（`current`、`game`、`gameState` など）
  - 主な関数名（`go`、`advanceWithinNodeOrNext`、`restart`、`loadFromSelectedSlot` など）
  - 新しく追加されたコンポーザブルの利用状況

- `apps/frontend/components/game/MessageWindow.vue`
  - props 名と型
  - emit 名

- `apps/frontend/components/game/StageCanvas.vue`
  - props 名と型

- `apps/frontend/components/game/MessageThemeModal.vue`
  - props / emit
  - 保存時に送信するフィールド名

- `apps/frontend/utils/gameState.ts`（存在する場合）
  - 公開している関数名と概要

- `packages/types/src/index.ts`
  - 追加・変更された型・インターフェース・定数の一覧

#### 主要な状態管理の対応表を更新
新しい状態管理が追加された場合は `## 主要な状態管理の対応表` に行を追加する。

#### 命名規則メモを更新
新しいパターンが増えた場合は `## 命名規則メモ` に追記する。

---

### docs/architecture.md

#### 更新が必要な箇所を以下の順にチェックする

**1. データモデル（`## データモデル` セクション）**

`apps/api/prisma/schema.prisma` を読み、以下を確認する:
- 新規モデルの追加 → セクションを追加
- 既存モデルへのフィールド追加・削除 → Prismaスキーマのコードブロックを更新
- enum の値変更 → enum 定義を更新

**2. API一覧（`## API 一覧` セクション）**

`apps/api/src/` 配下の `*.controller.ts` を走査し:
- 新しいエンドポイントを追加
- 削除されたエンドポイントを除去
- リクエスト/レスポンスの形式が変わっていれば更新

**3. URL/画面構成テーブル（`## URL / 画面構成` セクション）**

`apps/frontend/pages/` を走査し:
- 新規ページの追加 → テーブルに行を追加
- 削除されたページを除去

**4. アクセス制御（`## アクセス制御（AuthN / Z）` セクション）**

`apps/api/src/auth/supabase-auth.guard.ts` と各 controller の Guard 設定を確認し:
- 新しく認証必須になったエンドポイントを「要ログイン」に追加
- 公開になったエンドポイントを「公開（認証不要）」に移動

**5. 非同期処理（`## 非同期処理（BullMQ キュー）` セクション）**

`apps/api/src/queues/` と `apps/worker/src/` を確認し:
- 新しいキュー・ジョブが追加されていれば追記

**6. フロントエンド主要フロー（`## フロントエンド主要フロー` セクション）**

大きな処理フローの変更があれば更新する。
小さな変更（関数名の修正など）はfile-mapに任せてよい。

> ⚠️ architecture.md の ChangeLog に更新日と変更概要を追記すること:
> ```
> - YYYY-MM-DD: （変更内容の概要）
> ```

---

### docs/roadmap.md

`apps/frontend/pages/`、`apps/api/src/`、`apps/frontend/components/` を走査し、
以下の判断基準で完了済みタスクをマークする:

- `- [ ] xxx` のタスクに対応するページ・APIエンドポイント・コンポーネントが存在する → `- [x]` に変更
- 実装が途中の場合は `- [ ]` のまま残す
- 新機能が追加されていて roadmap に記載がない場合 → 適切なフェーズの `✅ 完了済み` に追記

---

## 更新時の注意事項

### やること
- コードの実態に合わせて**事実だけを書く**
- 「実装されているかどうか不明」な項目は `※ 要確認` を付けて残す
- 削除より**追記を優先**する（過去の経緯が消えないように）

### やらないこと
- 実装されていない機能を「完了済み」にしない
- コードを見ずに推測で書かない
- architecture.md の設計方針・規約セクションを勝手に変更しない
  （`## API/クライアント規約`、`## 用語ルール` など）

## 補足: Claude が特に必要としている情報

以下は Claude が実装指示を作る際に毎回参照する情報です。
この部分が古いと実装指示のファイル名・関数名がズレるため、**特に優先して最新化**してください。

1. **`play.vue` の状態変数・関数名** — バックログ・セーブ・演出などの新機能追加のたびに変わる可能性がある
2. **`packages/types/src/index.ts` の型一覧** — 新しい型を追加した際に必ず反映する
3. **`MessageThemeModal.vue` の保存フィールド** — テーマ関連機能を追加するたびに変わる
4. **`apps/api/prisma/schema.prisma` のモデル定義** — DB設計の基準点
5. **新規追加されたコンポーザブル・ユーティリティ** — 同じ処理を二重実装しないために必要
